# 03.5 — Data layer: SQLite → Firestore

**Status: ✅ Done.** Swapped the storage behind the tools from ephemeral SQLite
to managed **Cloud Firestore** — so tickets persist across restarts and are
shared across all Cloud Run instances. **Only `tools.py` changed;** `agent.py`
and every SKILL.md are byte-for-byte unchanged (the payoff of isolating data
access behind the tools).

---

## Why swap

On Cloud Run the container filesystem is ephemeral and per-instance:

| | SQLite (`/tmp`) | Firestore |
|---|---|---|
| Tickets after restart | ❌ lost | ✅ persist |
| Shared across instances | ❌ each isolated | ✅ one shared store |
| Fraud history (`get_refund_history`) | correct only within one instance | ✅ globally correct |

State belongs in an **external managed service**, not a file inside a stateless
container.

---

## What changed (and what didn't)

- **Changed:** `refund_agent/tools.py` internals — `sqlite3` → `google.cloud.firestore`.
  Collections `refund_orders` / `refund_tickets` (prefixed to stay isolated in a
  shared project). `orders` seeded from the bundled `orders.json` on first use.
- **Added:** `google-cloud-firestore` to `pyproject.toml` (pulled by the image build).
- **Unchanged:** the three tool signatures (`lookup_order`, `get_refund_history`,
  `create_escalation_ticket`), so `agent.py` and the SKILL.md never moved.

That is the whole point: **swap the storage layer, touch one file.**

## Setup (all CLI, no Firebase console)

```bash
# database already existed: (default), FIRESTORE_NATIVE, us-central1
gcloud projects add-iam-policy-binding linkhealth-care-2024 \
  --member=serviceAccount:refund-agent-dev@linkhealth-care-2024.iam.gserviceaccount.com \
  --role=roles/datastore.user
```

Redeploy adds a cost cap:

```bash
gcloud run deploy refund-agent --source=. --region=us-central1 \
  --max-instances=2 ...   # scale-to-zero when idle; capped when busy
```

## Proven — persistence + cross-instance sharing

1. A refund on Cloud Run (order 11111 → ESCALATE / IN_TRANSIT) created ticket
   `ESCA-…C6CAD0`.
2. A **separate local process** read that ticket back from Firestore.

→ Cloud Run wrote it, a different process read it = one shared, persistent store.
SQLite could not do this.

---

## Cost note

Firestore idle ≈ $0: we store kilobytes, far under the free tier (1 GiB storage,
50K reads / 20K writes per day free). Cloud Run scales to zero. `--max-instances=2`
bounds the worst case on the public endpoint.

---

## The one concept to remember

> Business logic never knows where data lives. Because data access is isolated
> in `tools.py`, migrating SQLite → Firestore touched **one file** — the agent
> and SKILL.md are untouched. Stateless container + external managed data is the
> cloud-native default.
